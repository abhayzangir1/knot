# Knot platform research register

**Last reviewed:** 2026-07-13
**Rule:** A verified fact is tied to a current primary source. Anything that
depends on a Slack workspace, administrator, account, or future platform change
is explicitly marked uncertain and must be capability-checked rather than
assumed.

## Uncertainties requiring capability checks

| Uncertainty | Why it matters | Required resolution |
| --- | --- | --- |
| Slackbot MCP option appears in the target app's App Settings | Controls whether an optional Slackbot MCP adapter can be configured | Check the target developer app at implementation time. If absent, do not expose or simulate the feature. |
| Agent View/AI entitlement in the target sandbox | Controls whether Agent View can be demonstrated | Confirm within the Slack Developer Program sandbox. Preserve shortcut/DM/modal/card equivalents regardless. |
| Slack app distribution status at submission | Controls whether RTS or Slack-hosted MCP are legally available | Treat the app as unlisted unless verified otherwise. Do not request/use either capability for the skeleton. |
| Target app still matches the reviewed manifest after dashboard changes | Slack dashboard toggles and reinstallation state can drift from the versioned manifest | Reimport/review `slack.json`, keep Agent experience off, reinstall after scope changes, and capture the final sandbox configuration in Phase-1 evidence. |
| Linear workspace approval and OAuth scopes | Admin policy controls installation and webhook access | Obtain from a real Linear test workspace only after the Slack gate; absent approval means the connector is not shown as connected. |
| Render account/database configuration and production domain | A Dockerfile and Blueprint cannot prove account-side resources, secret values, network placement, DNS, or a healthy public service | Keep local/container verification independent. Create the account-side service and database in one region, use the managed internal database URL, configure `/readyz`, replace the Slack request URL, reinstall the app, then capture live health and Slack evidence. |
| Slack credential lifecycle for distribution | The target app currently has token rotation disabled and no OAuth installation/refresh store | Keep the app limited to the controlled Phase-1 workspace. Do not distribute until a reviewed rotation/reinstall design and its tests satisfy D-035. |
| Devpost project form state and final submission acceptance | Login state and form validation are account-specific and can change | Recheck the official project form immediately before submission, grant the two required reviewer accounts sandbox access, and capture the accepted submission receipt before the deadline. |

## Rejected platform assumptions

- Slackbot MCP availability is not inferred from a blog announcement, a
  different workspace, or an old app. The current developer documentation takes
  precedence and requires capability detection.
- Slack's hosted MCP and RTS are not treated as a shortcut around Slack
  distribution restrictions.
- A model's interpretation of Slack content is not treated as an authoritative
  source of ownership, approval, privacy, action success, or closure.
- A successful HTTP dispatch is not treated as a successful external action
  without a provider receipt/reconciliation result.
- A passing local container or checked-in Render Blueprint is not treated as a
  live deployment, a healthy public endpoint, or a passed Slack gate.
- A non-rotating developer token is not treated as a distribution-ready
  credential lifecycle.

## Research refresh policy

Refresh this register before adding a Slack capability, changing the app
manifest/scopes, enabling MCP/RTS, adding a connector, deploying to a new
workspace, or submitting to Devpost. Record changed facts and their decision
impact in DECISIONS.md; do not silently overwrite an earlier conclusion.
