# Knot decision record

**Status:** architecture frozen on 2026-07-12
**Change rule:** Every material product, architecture, policy, security, data
retention, model, connector, or platform change gets a new numbered decision
below. A decision must state what changed, why, source evidence, uncertainty,
and required tests. Do not rewrite history to make an earlier decision appear
different.

| ID | Decision | Status | Rationale and consequence |
| --- | --- | --- | --- |
| D-001 | Knot is a Slack-native outcome coordinator, not a task manager or generic chatbot. | Accepted / frozen; closure wording clarified by D-029 | The product coordinates a bounded Outcome Contract and deterministic closure validation. The original phrase "verified closure" did not mean that Knot independently certifies external content; D-029 records the precise owner-attestation boundary. Knot excludes project, event, incident, routine, milestone, task, and connector-marketplace engines. |
| D-002 | Release outcome types are Request, Decision, Commitment, Handoff, and Other only. | Accepted / frozen | Four focused types make authority and closure deterministic; Other supports unclassified cases without speculative specialization. No additional type may be added before the skeleton gate passes. |
| D-003 | An Active shared outcome requires the eight-field Outcome Contract. | Accepted / frozen | Goal, exactly one accepted accountable owner, definition of done, next move, review point, evidence, participants, and privacy scope prevent Knot from becoming an activity list. Incomplete outcomes cannot activate, escalate, execute shared actions, or close. |
| D-004 | Participants, authorized audience, privacy scope, and connected systems are distinct models. | Accepted / frozen | Participants describe roles; ACL grants authorize access; privacy scope expresses disclosure intent; connected systems hold external state/permissions. This prevents participant-driven audience leakage. |
| D-005 | One accountable owner is mandatory; delegates and other roles are supported without transferring accountability. | Accepted / frozen | A delegate can act under policy, but only a separately authorized ownership-transfer transaction changes the accountable owner. |
| D-006 | Application services are the sole domain authority. | Accepted / frozen | Slack handlers and MCP are thin adapters that create verified ActorContext and call the same services. Neither accesses repositories directly or copies lifecycle/policy logic. |
| D-007 | Slack ingress acknowledges immediately, with a minimal durable receipt exception. | Accepted / frozen | Slack's acknowledgement budget requires fast response. A tiny idempotent receipt may be stored before ack to prevent lost work; domain writes, model inference, and connector calls are asynchronous. |
| D-008 | Every model-extracted contract field requires human confirmation/edit before shared activation. | Accepted / frozen | The model may produce candidates with evidence labels only. The creator confirms all fields/participants/privacy scope, and the proposed owner accepts accountability. |
| D-009 | Authority is deterministic and separates duties by default. | Accepted / frozen | Identity mapping, audience authorization, type authority, owner rules, and configurable self-approval restrictions are policy-service decisions. Consequential self-approval is denied when requester, executor, owner, or sole beneficiary overlaps, except for explicit audited policy exceptions. |
| D-010 | Approval binds an immutable, versioned action plan. | Accepted / frozen | An approval must bind tenant, actor, outcome/contract version, before-state version, evidence snapshot, policy version, plan hash, expiry, and idempotency key. Staleness invalidates it. |
| D-011 | Actions are classified before approval; compensation is version-checked. | Accepted / frozen | Reversible actions have exact restore; compensatable actions have a disclosed counter-action; irreversible actions say so. External compensation checks current version/ETag and stops at manual resolution on conflict. |
| D-012 | The walking skeleton uses a real reversible Slack card update. | Accepted / frozen | The app-owned card has a stored before payload, Slack receipt, exact approved update, and rollback proof. It is a stronger gate than a mock action. Delivered DMs are compensatable, not falsely reversible. |
| D-013 | No integration or broad capability is built before the walking skeleton passes. | Accepted / frozen | Linear, MCP transport, Slackbot MCP, RTS, Slack-hosted MCP, App Home/Agent View enrichment, monitoring, expanded infrastructure, and more types are blocked by PLANS.md Phase 1. |
| D-014 | Knot Core MCP will be an authenticated Streamable HTTP adapter after the gate. | Accepted / frozen | It validates Origin/input/output, derives actor context, invokes application services, and preserves audit parity. It never becomes a second domain implementation. |
| D-015 | Slackbot MCP is optional and capability-gated. | Platform premise superseded by D-032; product policy retained | When recorded, workspace access was treated as still rolling out. The durable product decision was that Slackbot MCP could not block the core release. D-032 records the corrected availability premise without weakening capability detection. |
| D-016 | RTS and Slack-hosted MCP are not release dependencies. | Accepted / frozen | Slack restricts both to eligible Marketplace/directory-published or internal apps. Knot's unlisted sandbox path remains complete without them. |
| D-017 | Linear is the only external connector in the closed product. | Accepted / frozen | It is built fully after the skeleton: OAuth, encrypted credentials, signed webhooks, preview/approval, receipts, reconciliation, and compensation. No placeholder or fake connector UI is permitted. |
| D-018 | Unknown external state remains explicit. | Accepted / frozen | Internal effects are exactly-once through idempotency. External effects require provider idempotency/receipts or become Unknown until reconciled; Knot never reports success by inference. |
| D-019 | Privacy-minimizing audit retention is required. | Accepted / frozen | Retain structured facts and source references where possible, not raw private Slack corpora. Deletion removes/redacts private content but preserves a minimal non-sensitive integrity tombstone. |
| D-020 | evaluation-policy-v1 governs release quality. | Accepted / frozen | Versioned fixtures/evaluators/models/policies set thresholds. Known unsupported critical claims, authorization bypasses, duplicate internal effects, and invalid closure have zero tolerance; other quality claims use measured thresholds. |
| D-021 | Accessibility has concrete acceptance criteria. | Accepted / frozen | Every core flow needs complete fallback text, text-based status, specific controls, labels/hints/errors, keyboard paths, mobile completion, and manual screen-reader checks. External pages require axe, keyboard, focus, zoom, and 320 CSS-pixel verification. |
| D-022 | Documentation is implemented beside tested behavior, except the foundation records. | Accepted / frozen | Before the skeleton, only AGENTS.md, PLANS.md, ARCHITECTURE.md, OUTCOME_LIFECYCLE.md, THREAT_MODEL.md, RESEARCH.md, and this record are created. Other docs evolve with passing implementation. |
| D-023 | Role-specific private Knot Messages replace one private Slack channel per outcome. | Accepted / frozen clarification | Per-outcome channels create unbounded workspace clutter and expose static controls to the wrong people. Knot stores one canonical status card in the owner’s private Messages tab; if another person owns the contract’s next move, only they receive its separate single-action card. Requester/reviewer updates are view-only until a role-bound request is needed. Closure sends detailed evidence summaries only to principals with both view and evidence access. Knot retains its database audit record and does not automatically delete Slack channels or messages. Tests must prove selected-role authority, no shared self-confirmation through a personal card, private single-message self-confirmation, and audience-filtered delivery. |
| D-024 | The contract modal must expose every already-frozen outcome type. | Accepted / implementation correction | The domain had deterministic type-specific closure rules but the Slack form silently persisted every outcome as `request`. That is a misleading partial release, not a safe scope reduction. The modal now offers Request, Decision, Commitment, Handoff, and Other; no new type or specialized engine was added. Tests prove the selected value survives binding and each type accepts only its allowed closure-evidence kind. D-013 still blocks any new type or external integration until the live gate passes. |
| D-025 | Outcome roles may be assigned only to active human Slack members. | Accepted / security correction | Slack's user picker can include Slackbot and app users, but they cannot meaningfully accept accountability or provide independent approval. Before consuming the single-use preview or creating domain state, the Slack adapter calls `users.info`, verifies the exact selected user ID, and rejects deleted, bot, or app users. This requires the narrow `users:read` bot scope; no email scope is requested or used. Tests cover active users, bots, app users, deleted members, missing users, and identity mismatch. |
| D-026 | State-changing Slack commands use one bounded durable-receipt transaction before acknowledgement. | Accepted / reliability correction | The lost-work exception in D-007 is implemented as one workspace-bound `outbox_jobs` insert with a content hash and unique delivery key. The synchronous path does not create an outcome, resolve/create a principal, call Slack, or invoke a domain service. The worker resolves ActorContext and rechecks tenant and authority. Tests cover exact retries, payload collisions, stale leases, terminal redaction, and RLS. Live p95/p99 acknowledgement measurements remain a Phase-1 gate. |
| D-027 | The configured bot token is bound to exactly one Slack workspace at startup. | Accepted / tenant-boundary correction | `auth.test` supplies the authenticated `team_id`. Every signed shortcut, view, and action payload must match it before a receipt is enqueued; the worker independently verifies that the resolved internal workspace matches the job workspace. A mismatched or malformed identity fails closed. This is the intentional single-workspace skeleton boundary, not multi-workspace OAuth. |
| D-028 | Recoverable Knot DMs use opaque Slack message metadata and retry-only history reconciliation. | Accepted / delivery correction | Slack has no general `chat.postMessage` idempotency key. Knot posts a non-sensitive delivery ID in message metadata. A first attempt avoids `conversations.history`; only an ambiguous durable-job retry scans the latest supported 15 messages with metadata and suppresses a duplicate. This respects current history rate-limit constraints while preserving a recoverable user-visible effect. The identifier contains no outcome text or authority. |
| D-029 | Phase-1 closure is an authorized owner attestation, not independent content verification. | Accepted / truthfulness correction | Knot validates actor authority, current outcome state, evidence type, freshness label, HTTPS locator syntax, and contract rules. It does not fetch or certify the external page contents. Cards, fallback text, audit reasons, and tests must preserve that distinction; no successful action alone closes an outcome. |
| D-030 | The Phase-1 Slack manifest is exact and keeps conversational agent controls disabled. | Superseded in scope list by D-056 | The original scope list omitted Slack's required `commands` grant for the declared message shortcut. Messages remains enabled and read-only; Home and Agent experience remain disabled because Phase 1 has no `message.im` conversational handler. The app creates no outcome channels and requests no channel-management or email scope. Reinstall is required after manifest/scope changes. |
| D-031 | Durable and audit failure records retain stable codes, not private provider or Slack text. | Accepted / privacy correction | Queue payloads are redacted after completion or terminal failure, payload hashes preserve collision detection, logs serialize errors to safe type/code/status fields, and audit failure details store stable error codes. Slack action receipts retain only method, target identifiers, and confirmed status/version needed for reconciliation. Tests and log review must reject raw tokens or private source content. |
| D-032 | Slackbot MCP support is documented but remains installation-capability-gated. | Platform premise corrected by D-048; product policy retained | Slack's June 2026 announcement publicly documents Slackbot MCP Client support, while the implementation guide still describes developer access as rolling out. Workspace settings, installation eligibility, and administrator approval can differ, so Knot must detect the capability in the target installation and remain fully usable without it. This does not authorize MCP implementation before the Phase-1 walking-skeleton gate. |
| D-033 | A bounded Docker/Render package may support the live Phase-1 Slack gate without expanding product scope. | Accepted; database placement superseded by D-051 | A stable public TLS receiver is necessary to exercise Slack HTTP interactivity. The checked-in Dockerfile, Compose service, and Render Blueprint package only the existing modular receiver and PostgreSQL dependency; they add no surface, integration, outcome type, or speculative service. Docker base images are digest-pinned, the runtime is non-root, and `/healthz` and `/readyz` expose process/readiness state. Account-side credentials, database state, public domain, and live health remain unverified until captured in Phase-1 evidence, so packaging is not a deployment-success or publishability claim. D-051 replaces the original Render-managed database placement. |
| D-034 | Migration serialization and isolated PostgreSQL tests are mandatory runtime safeguards. | Accepted / database-reliability correction | Startup migrations take one session-level PostgreSQL advisory lock on a dedicated connection and destroy that session on exit, preventing concurrent replicas from racing the migration set. `npm run test:postgres` requires `TEST_DATABASE_ADMIN_URL`, creates a unique database, strips Slack credentials from child processes, runs migrations and tests there, then terminates connections and drops it even after failure. Local Compose exposes PostgreSQL only on loopback port 5433. Clean-container health, two simultaneous migrators, isolated-suite cleanup, RLS, and runtime-database non-interference are required evidence. |
| D-035 | Disabled Slack token rotation is a temporary single-workspace Phase-1 installation boundary, not a distribution-ready setting. | Accepted / credential-lifecycle boundary | The skeleton accepts one manually configured bot token and binds it to the `team_id` returned by `auth.test`; it does not yet implement an OAuth installation store or refresh-token lifecycle. Slack documents manifest-enabled token rotation and refresh behavior. Therefore `token_rotation_enabled: false` is permitted only for the current controlled developer workspace. Public or multi-workspace distribution is blocked until an accepted credential-rotation and reinstall design, encrypted installation storage, revocation handling, and tests exist. Reinstallation after a manually revoked non-rotating authorization is an operational recovery, not a substitute for the distribution design. |
| D-036 | Phase-1 database changes use reviewed immutable SQL without shipping `drizzle-kit`. | Accepted / dependency-security correction | Runtime migration execution uses `drizzle-orm` and the committed ordered SQL files; the `drizzle-kit` CLI was used only for optional generation/studio tooling and pulled a development-server `esbuild` advisory with no safe stable upgrade path. Knot removes that unused CLI instead of accepting the advisory or forcing a breaking downgrade. New schema work must add and review an immutable SQL migration, then pass fresh-database, concurrent-migrator, RLS, and isolated PostgreSQL tests. |
| D-037 | The test runner uses one worker thread and no file parallelism. | Accepted / test-reliability correction | On constrained Windows hosts, Vitest's default fork-per-core scheduling exhausted virtual memory after otherwise passing assertions. One worker thread preserves isolation, keeps the full database suite deterministic, and avoids false-green runs caused by worker teardown errors. Both the unit run and isolated PostgreSQL run must exit cleanly; slower execution is an accepted release-gate cost. |
| D-038 | A declined proposed-owner request is recovered by creator-authorized reassignment, not ownership transfer. | Accepted / lifecycle-recovery clarification | The declined request remains inactive in Clarified, so no accountable ownership exists to transfer. Only the original creator with explicit edit permission may submit a different proposed owner through a complete reconfirmed contract, provenance set, and authorized audience. The service increments both outcome and contract versions, revokes any stale active delegation, resets acceptance to Pending, audits the reason and old/new owner IDs, and returns to AwaitingOwnerAcceptance. The former proposed owner cannot accept the new request; only the newly named owner can activate it. |
| D-039 | Durable action-preview jobs bind one explicit action-plan ID and idempotency key. | Accepted / duplicate-effect correction | The job chooses and persists opaque binding values before application-service work. A retry supplies the same binding; the service returns the existing plan only when workspace, outcome, creator/executor, and idempotency key match, otherwise it fails closed. A first attempt persists a plan with the supplied UUID and key. This prevents a retry after partial delivery from silently creating another internal action plan while keeping authorization and immutable exact-plan checks in the application service. |
| D-040 | Recoverable action reconciliation is distinct from terminal unknown state. | Accepted / execution-state clarification | An inconclusive dispatch or temporarily unavailable card inspection leaves the plan in Dispatching and returns `action_reconciliation_pending`, allowing a bounded worker retry that must inspect the exact card state before dispatching again. An observed card matching neither the exact before nor after state moves the plan to Unknown and returns `action_state_unknown`; it is terminal pending manual reconciliation. Exact after state finalizes Applied without redispatch, while exact before state permits one safe dispatch retry. |
| D-041 | Evidence projections reconcile removals during contract correction. | Accepted / privacy-retention correction | `evidence_references` is a current-state projection, not an immutable audit archive. Each outcome update deletes projection rows whose evidence keys are absent from the corrected contract before upserting the current set, so a removed private permalink cannot survive in a stale projection. Append-only audit events retain non-sensitive correction integrity without retaining the removed locator. PostgreSQL tests must prove both replacement and later deletion behavior. |
| D-042 | Recovery and management controls are role-bound, versioned, and complete in Slack. | Accepted / lifecycle and authorization correction | A declined owner request gives only the creator a reconfirm-and-reassign or cancel path. Active owner cards expose owner actions; a separate requester card exposes correction and deletion; the named next-move person receives the update action; delegates see only controls represented by their active stored delegation; only the canonical owner closure card can reopen. A correction atomically replaces the contract, provenance, and audience, and deletion is limited to the creator or an audience-authorized workspace administrator. Deletion removes private content before projecting a generic receipt so Slack does not receive another copy of deleted content. Durable retries reuse stable command time/identity and do not duplicate delegations, corrections, reassignment, or reopening. Tests cover role-specific cards, complete recovery schemas, reconfirmation, audience replacement, retry idempotency, deletion authority, and retained redacted audit integrity. |
| D-043 | Invalid installation identity is acknowledged and rejected before durable work. | Accepted / ingress reliability correction | Signature verification remains the receiver boundary. After minimal signed-payload parsing, every view and action compares the payload team with the bot-token installation. A mismatch or malformed identity receives an immediate empty acknowledgement, writes no durable command, and emits only a safe structured warning. This preserves the three-second interaction contract without weakening tenant isolation. Tests prove the negative path acknowledges exactly once and returns no actor identity. |
| D-044 | The hackathon submission targets New Slack Agent and cannot rely on the Phase-1 skeleton alone. | Accepted / current submission constraint | Devpost's current rules require at least one Slack AI capability, MCP server integration, or RTS use and set the deadline at July 13, 2026, 5:00 p.m. PDT. The Organizations track additionally requires a Slack Marketplace submission and five active workspaces, which the controlled single-workspace credential boundary in D-035 intentionally does not satisfy. The fastest truthful path is New Slack Agent: first pass and record the live Phase-1 gate, then implement and demonstrate the already-frozen authenticated Knot Core MCP adapter and Slack-native agent behavior. Do not claim eligibility from a shortcut-only skeleton, a mock MCP endpoint, or a merely configured but unused capability. |
| D-045 | The final container excludes package-management tooling that the receiver does not use. | Accepted / runtime attack-surface correction | A current Docker Scout scan found high-severity CVE-2026-12151 in `undici` 6.26.0 bundled inside the Node base image's global `npm`, while `npm audit` and Knot's application dependency graph were clean. The receiver starts with `node`, so the runtime stage now removes global `npm` and `npx` after installing locked production dependencies. The rebuilt image retains the application and acknowledgement probe, runs non-root, passes health checks, and scans with zero critical, high, medium, or low package findings. Build-stage tooling remains outside the final image. |
| D-046 | Phase-1 deployment hardening must fail early and shut down completely. | Accepted / runtime reliability correction | Compose binds PostgreSQL and Knot only to loopback; public access requires an explicit TLS path. Startup validates the non-sensitive shape of the Slack signing secret and bot token before opening ingress. Shutdown stops HTTP first, drains background and durable work, attempts database cleanup even after an earlier failure, and sets a failing exit status for incomplete cleanup. The deployment probe requires unsigned, forged-signature, and correctly signed stale-replay requests to return HTTP 401 before measuring acknowledgement latency. Tests cover credential shape, cleanup order/failure, packaging boundaries, and probe thresholds; a live SIGTERM run must exit zero and restart ready. |
| D-047 | Deduplicated read-only Slack retries must terminate every visible preparing surface. | Accepted / replay UX correction | A duplicate shortcut or exact-review request has no second domain effect, but silently returning could leave a second Preparing modal indefinitely. Duplicate shortcut/review surfaces now close with explicit no-duplicate guidance, while a duplicate status check returns a private already-processing-or-complete receipt. This preserves ingress idempotency without implying that the original request succeeded. Tests lock the truthful feedback, and the live replay gate must confirm no stuck modal or duplicate outcome, check, or approval. |
| D-048 | Target-workspace capability evidence, not a global-availability claim, authorizes Slackbot MCP setup. | Accepted / platform-fact correction | The current Slackbot MCP Client guide says the feature is still rolling out to developers and tells developers to detect the MCP Servers entry in App Settings. Knot's target app displayed that entry on 2026-07-13, so the target sandbox can proceed after the Phase-1 gate. Other installations remain capability-gated. Slack identity auth is the only acceptable target mode because signed requests carry the Slack user/team identity needed to derive ActorContext; an unauthenticated, shared-identity MCP surface cannot expose outcome data. |
| D-049 | The submission deployment uses explicit Render Free instances and is labeled a hackathon sandbox. | Superseded in part by D-051 | The user requires a zero-cost Render deployment. Render's Blueprint schema accepts `plan: free` for both web services and PostgreSQL, so the checked-in Blueprint pins both and automated packaging tests reject paid instance types. Render also documents an approximately one-minute wake after 15 idle minutes, a 30-day/1 GB/no-backup Free PostgreSQL boundary, and says Free instances are not for production. Live Blueprint validation additionally proved that `maxShutdownDelaySeconds` is unsupported on Free, so the Blueprint omits it and a regression test locks that constraint. D-051 retains the Free web service but replaces the expiring Render database. |
| D-050 | A Free external availability monitor protects judge access without becoming product monitoring. | Superseded in probe target by D-051 | Official rules schedule judging through August 6 and winners around August 11. Render recommends external probes, and UptimeRobot's Free five-minute interval is shorter than Render's 15-minute idle period. This remains an infrastructure availability probe only: it reads no Slack content, creates no outcome, and does not bypass the frozen product-monitoring boundary. D-051 changes the target from `/readyz` to `/healthz` so keep-warm traffic does not continuously consume external database compute. |
| D-051 | Non-expiring Neon Free PostgreSQL replaces expiring Render Free PostgreSQL; liveness and readiness probes are separated. | Accepted / region detail superseded by D-053; supersedes the database and probe targets in D-049 and D-050 | Winner timing is expressly “on or around” August 11, so a database expiring around August 12 is not a defensible continuity margin. Neon Free has no time limit or card requirement and supports PostgreSQL 17 in Oregon, standard TLS connections, scale-to-zero, and bounded free allowances. The Render Blueprint therefore creates only one Free Oregon web service and prompts for a secret Neon `DATABASE_URL`; it cannot create a paid or expiring Render database. Knot uses Neon's Direct connection because its migration lock is session-scoped. Render and UptimeRobot probe `/healthz` so keep-warm traffic does not consume continuous Neon compute; `/readyz` remains the end-to-end gate before tests and after alerts. This operational provider choice adds no product surface, connected-system behavior, or mock data. |
| D-052 | Phase-1 creation must not present generic template text as extracted outcome intelligence. | Accepted / truthfulness and UX correction | The previous shortcut preview copied the real source message into the goal but also prefilled generic definition-of-done, next-move, and two-day review values. Those values were neither model-produced nor evidence-derived and could be mistaken for meaningful extraction. Until a real, evaluated extraction service is authorized after the walking-skeleton gate, Knot leaves those required fields and the review trigger unselected for explicit human entry. The source-grounded goal and Slack permalink remain real evidence. Tests require the creation modal to omit fabricated initial values while correction/reassignment continues to repopulate actual stored contract data. |
| D-053 | Match the Render service to the verified Neon Ohio project instead of recreating the empty database. | Accepted / deployment-region correction | The newly created Neon project exposed an `aws-us-east-2` direct endpoint, which Neon's current regions documentation identifies as Ohio. Render's current regions and Blueprint documentation support an Ohio web service through `region: ohio`. Because the project is empty and both providers offer the same region, aligning the not-yet-created Render service preserves the intended low-latency same-region deployment without data migration, added cost, product-scope change, or deadline risk. The Blueprint, packaging test, deployment guide, research record, plan evidence, and submission checklist must all name Ohio consistently. |
| D-054 | A contract preview permits one careful one-hour review and one durable creation command. | Accepted / live-gate reliability correction | The first stable-host test proved that ten minutes was too short to review and confirm every Outcome Contract field; two later submissions of the same modal also produced separate durable jobs because their Slack view hashes differed. The durable context stores no source text, is scoped to the verified workspace and creator, grants no authority, and is single-use, so a one-hour expiry is a bounded usability correction. Contract creation now binds the preview reference, tenant, and actor to the fixed nonce `once` and one deterministic outcome ID, preventing repeated clicks or differently hashed Slack deliveries from creating duplicate jobs or outcomes. Tests lock the lifetime, expiry, single-use behavior, and stable binding. The live gate must be repeated after deployment. |
| D-055 | PostgreSQL connection modes that pg 8 aliases to strict verification are normalized to `verify-full`. | Accepted / TLS future-proofing | The live Neon logs warned that `prefer`, `require`, and `verify-ca` currently behave as `verify-full` but will adopt weaker libpq semantics in the next pg major version. Knot explicitly rewrites those three modes to `verify-full` before every application or migration pool is created, preserving the current certificate and hostname verification behavior, retaining other parameters such as channel binding, and removing the warning. Connections without one of those modes are unchanged. Unit tests lock the transformation. |
| D-056 | The message shortcut's `commands` scope is required and channel-management grants remain prohibited. | Accepted / live scope correction | A live contract-creation job failed closed at `users.info` because the installed token still held the old grant set. During the correction, the checked-in manifest was found to omit `commands` even though the live app already had it. Slack's current shortcut guide and scope reference explicitly require `commands` for shortcuts. The exact Phase-1 bot scopes are therefore `chat:write`, `commands`, `im:write`, `im:history`, and `users:read`. Knot exposes no slash command and creates no outcome channels, so `groups:write` and `groups:write.invites` are unnecessary and remain prohibited. Scope changes require reinstall and a fresh live gate. |

## Evidence and uncertainty references

- Platform evidence is maintained in RESEARCH.md.
- D-007 is grounded in Slack's acknowledgement requirement and must be
  rechecked when changing ingress.
- D-014 is grounded in the MCP Streamable HTTP specification and must be
  rechecked when selecting the final protocol version/library.
- D-032 and D-016 are intentionally capability/distribution constrained. They
  may not be relaxed based on an expectation, blog post, or another workspace.
- D-033 is only a deployment-enablement clarification. It does not waive the
  live Phase-1 evidence gate or permit the product work blocked by D-013.
- D-035 makes the current app intentionally non-distributable. Any distribution
  work requires a new accepted credential-lifecycle decision and current Slack
  platform evidence.
- D-044 selects the New Slack Agent submission path. It does not waive the
  Phase-1 live gate or authorize a fake or unused qualifying technology.
